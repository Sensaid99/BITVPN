# Нагрузка на сервер при работе мини-аппа

Оценка при **1000 пользователей**, одновременно открывших мини-апп, и что сделано для снижения нагрузки.

---

## Какие запросы идут при открытии приложения

| Действие пользователя | Запрос к вашему API | Внешние вызовы с сервера |
|------------------------|---------------------|---------------------------|
| Открыл мини-апп       | GET /plans, POST /me | Happ (add-install или get_install_stats), БД |
| Загрузка главного экрана | GET /my-ip, GET /servers-status | ip-api.com (1 раз на IP), при SERVER_LIST — N запросов к вашим серверам |
| Нажал «Замерить скорость» | GET /speed-test-file (1 МБ) | — |

Без кэша при 1000 пользователей за минуту было бы:
- 1000 запросов к ip-api.com → **лимит бесплатного тарифа ip-api.com ~45 req/min**, запросы начнут падать.
- 1000 × (число URL в SERVER_LIST) исходящих запросов с вашего API к списку серверов (например 50 × 1000 = 50 000 запросов) → большая нагрузка и риск упереться в лимиты.

---

## Что сделано для снижения нагрузки

### 1. Кэш по IP для «страна + флаг» (my-ip)

- **Кэш:** результат по каждому IP хранится **10 минут**.
- **Эффект:** при 1000 пользователей с разными IP в первую минуту будет до 1000 запросов к ip-api.com (всё равно много для бесплатного лимита). При повторных заходах тех же пользователей или при общем NAT (один IP на много пользователей) запросов к ip-api.com будет **значительно меньше** (один раз на IP за 10 мин).
- **Рекомендация:** при росте до тысяч пользователей рассмотреть платный геосервис с большим лимитом или кэш в Redis с TTL 10–15 мин.

### 2. Кэш для «работающие серверы» (servers-status)

- **Кэш:** один результат на **60 секунд** для всех пользователей.
- **Эффект:** список серверов пингуется **не чаще раза в минуту**, а не при каждом открытии главного экрана. 1000 пользователей получают один и тот же закэшированный ответ. Исходящих запросов к вашим серверам: **N раз в минуту** (N = число URL в SERVER_LIST), а не N × 1000.

### 3. Замер скорости (speed-test-file)

- Запрос только **по нажатию** пользователя, не при каждой загрузке.
- Один замер = 1 МБ трафика. Даже если 100–200 человек в минуту нажмут «Замерить», это 100–200 МБ исходящего трафика и 100–200 быстрых ответов — для одного сервера обычно приемлемо.

---

## Ориентировочная нагрузка при 1000 пользователей

- **Ваш API (uvicorn):** несколько тысяч запросов в минуту (plans, me, my-ip, servers-status) — реально для одного воркера, если запросы не все в одну секунду. При пиках лучше 2–4 воркера (uvicorn --workers 2) или балансировка.
- **БД:** запросы на /me (чтение пользователя и подписки) — нормальная нагрузка для PostgreSQL при индексах.
- **Внешние сервисы:** ip-api.com при большом числе уникальных IP может упираться в лимит (см. выше); Happ — по их лимитам.
- **Трафик:** в основном 1 МБ на каждый замер скорости; остальные ответы маленькие.

---

## Итог

- Для **сотен** одновременных пользователей кэш my-ip и servers-status сильно снижает нагрузку и риск упора в лимиты.
- Для **1000+** пользователей с разными IP стоит планировать платный гео-API или более длинный кэш по IP, несколько воркеров uvicorn и мониторинг (нагрузка, ошибки, лимиты внешних API).
