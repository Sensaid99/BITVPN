# Официальный URL Happ API

По документации Happ:

- **Базовый URL API:** [https://happ-proxy.com](https://happ-proxy.com/)  
- Все эндпоинты — относительные пути к этому URL.
- Формат ответа: `application/json`.
- Результат выполнения — в полях `rc` и `msg` в теле ответа (HTTP 200).

Используемые в проекте эндпоинты:

| Метод и путь | Назначение |
|--------------|------------|
| GET `/api/add-install` | Создать лимитированную ссылку (выдача подписки пользователю). |
| GET `/api/list-install` | Список лимитированных ссылок (счётчик «Подключено» в мини-аппе). |
| GET `/api/add-domain` | Привязать домен подписки к provider (при необходимости). |

В `.env` задаётся **один** базовый URL: `HAPP_API_URL=https://happ-proxy.com`. Опционально для счётчика можно задать отдельный `HAPP_LIST_INSTALL_URL`, если у провайдера иной хост для list-install.

## Provider ID (учёт устройств и проверка подписки)

По документации Happ приложение не чаще раза в сутки запрашивает `https://check.happ-proxy.com/provider?id={ProviderID}` (с хешем домена, HWID, ОС). Чтобы счётчик устройств и привязка к вашему аккаунту работали, **ProviderID** (= `HAPP_PROVIDER_CODE`) нужно передать в подписку. Способы по документации:

1. **В теле подписки (комментарий)** — `#providerid {ProviderID}` в начале контента.  
2. **В HTTP-заголовке ответа** — `providerid: {ProviderID}`.

В проекте при проксировании подписки по эндпоинту **GET /sub/{install_code}** (когда API отдаёт контент напрямую, а не редиректом) делается:

- в **заголовок** ответа добавляется `providerid: <HAPP_PROVIDER_CODE>`;
- в **тело** подписки в начало добавляется строка `#providerid <HAPP_PROVIDER_CODE>` (в декодированном base64-контенте перед перекодированием).

Так приложение Happ получает ProviderID и может корректно учитывать устройства и обращаться к check.happ-proxy.com.
