# Разбор ошибок и план исправлений (VPN BOT + мини-апп + Happ)

Документ для поэтапной работы: как прислать логи, что проверить по каждому пункту, где смотреть в коде.

---

## Как прислать логи (удобно для Cursor / поддержки)

### Общие правила
- **Не вставляй** целиком `.env`, токены бота, ключи ЮKassa/Happ, пароли БД. Достаточно: «переменная задана / не задана».
- Один сценарий = один блок лога (например только `/start`, только открытие мини-аппа).
- Укажи: **где** воспроизводится (ПК Windows / Android / iOS), **браузер** или **Telegram**, **время** по часам сервера (если есть).

### 1) Бот на VPS (`vpn-bot`)
На сервере по SSH:

```bash
# последние 200 строк
sudo journalctl -u vpn-bot -n 200 --no-pager

# с фильтром по start / ошибкам
sudo journalctl -u vpn-bot -n 400 --no-pager | grep -E "start_command|ERROR|Exception|Traceback|TelegramError|TimedOut"
```

Скопируй вывод в файл или в чат. Если был **Traceback** — обязательно **все строки** от `Traceback` до конца.

### 2) API мини-аппа на сервере (`miniapp-api`)
```bash
sudo journalctl -u miniapp-api -n 200 --no-pager
sudo journalctl -u miniapp-api -n 400 --no-pager | grep -iE "error|exception|traceback|500|502"
```

### 3) Nginx (если 502/504/таймауты)
```bash
sudo tail -n 100 /var/log/nginx/error.log
sudo tail -n 100 /var/log/nginx/access.log
```

### 4) Мини-апп в браузере / Telegram WebView
- **ПК Chrome:** F12 → **Network** → включи **Preserve log** → воспроизведи проблему → экспорт HAR (опционально) или скрин списка запросов с **красными** строками.
- **Console** — скопируй красные сообщения (ошибки JS / CORS / failed to fetch).

### 5) Локально на ПК (если бот крутится не на VPS)
- Файлы в папке проекта **`logs/`** (если включено файловое логирование).
- Вывод терминала, где запущен `python run.py`.

### 6) Версии для контекста
- Версия **Happ** на телефоне (из «О приложении»).
- Версия **Telegram** (приложение).
- Коммит/дата деплоя: на сервере `cd /opt/vpn-bot && git log -1 --oneline` (или ваш путь).

## Пункт 1: Бот не реагирует на `/start`

**Симптом:** в чате нет ответа на `/start`.

**Что уже чинили в коде (см. журнал сессий):** таймауты Telegram HTTP, пул соединений `connection_pool_size`, `send_message` без `wait_for`, логи `send_message begin` / `ok`.

**Что прислать:**
- Фрагмент `journalctl -u vpn-bot` с момента нажатия `/start` (строки `start_command`, `before_db`, `sending reply`, `send_message ok` или `TelegramError`).
- Значение на VPS (без секрета): задан ли **`TG_HTTP_TIMEOUT`**, **`TG_HTTP_POOL_SIZE`** в `.env`.

**Где в коде:** `bot/handlers/main.py` (`start_command`), `bot/main.py` (`HTTPXRequest`).

**Этап проверки:** логи → сеть до `api.telegram.org` → актуальный `git pull` и рестарт `vpn-bot`.

## Пункт 2: На ПК в Happ список серверов нормальный, на телефоне — один сервер и «старый»

**Симптом:** разный контент подписки (кол-во нод / старая нода) между платформами.

**Возможные причины:**
- Разный **User-Agent**: 3x-ui / subscription endpoint иногда отдаёт **разный JSON** для «мобильного» и «десктопного» клиента.
- **Кэш** Happ / старая подписка не обновлена (кнопка обновления в Happ).
- Запрос к **разным URL** (редирект `/sub/КОД`, `HAPP_SUBSCRIPTION_URLS`, прокси через `api_miniapp`).

**Где в коде / конфиге:**
- `HAPP_SUBSCRIPTION_FETCH_UA` в `.env` — явный UA при сборке подписки на бэкенде.
- `api_miniapp.py` — сборка `/sub/...`, мерж нескольких `HAPP_SUBSCRIPTION_URLS`.
- `bot/utils/subscription_card.py`, выдача ссылок в боте.

**Что прислать:**
- Точная **ссылка подписки** (можно замазать токен), с которой сравниваешь ПК и телефон.
- Один **`curl -sS -A "..."`** с сервера на URL подписки с двумя разными `-A` (десктоп vs мобильный) и сравнение размера/первых строк ответа (без секретов).

**Этап:** зафиксировать UA → выровнять ответ 3x-ui или принудительно **один** UA в бэкенде при проксировании.

## Пункт 3: Мини-апп то грузится, то нет; иногда долго

**Симптом:** нестабильная загрузка UI, долгий белый экран.

**Возможные причины:**
- Таймауты до **`MINIAPP_API_URL`** (мобильный интернет, DNS).
- **Vercel** холодный старт / ошибка edge.
- **CORS** или смешение `?api=` (запросы уходят не туда).
- Тяжёлый **`index.html`** / 3D / шрифты.

**Что прислать:**
- С телефона: скрин **Network** невозможен — опиши: «висит на … сек», есть ли текст ошибки в приложении.
- С ПК: F12 Network — какой запрос **красный** (`/api/miniapp/init` или другой), код ответа, время.
- `journalctl -u miniapp-api` за тот же интервал.

**Где в коде:** `webapp/index.html`, `api_miniapp.py` (таймауты, CORS), nginx при прокси.

**Этап:** воспроизвести один запрос → увеличить таймауты клиента / оптимизировать первый экран / CDN.

## Пункт 4: Сайт недоступен без VPN и с мобильного интернета

**Симптом:** лендинг / домен блокируется провайдером или недоступен из РФ без VPN.

**Идеи обхода (продуктовые, не «обход блокировок» в незаконном смысле):**
- Зеркало на **другом домене** / поддомене с другим DNS.
- Отдача **статического лендинга** с **Vercel** (уже есть мини-апп) — вынести маркетинговую страницу туда же или на отдельный проект.
- Ссылка «сайт» в боте вести на **Telegram-канал** / **мини-апп** как основной вход.
- Для проверки доступности: **Cloudflare** (если политика позволяет), другой хостинг регион.

**Этап:** решить, что считаем «сайтом» (`website/` vs bitecosystem.ru) → дублирование или перенос на менее фильтруемый хост.

## Пункт 6: После добавления подписки в Happ — редирект в приложение; мини-апп ломается («обновите мини-апп»); ссылка ведёт на Vercel

**Симптом:** deep link / `happ://` / возврат из Happ ломает состояние WebView; ошибка про обновление мини-аппа; нежелательный уход на Vercel.

**Связано с:** обработка **`visibilitychange`**, сброс промисов загрузки API, `WEBAPP_URL`, параметр `?api=`, открытие ссылок через `openTelegramLink` / `tg.openLink`.

**Где в коде:** `webapp/index.html` (уже были правки под возврат из Happ), `api_miniapp.py`, настройки меню бота в `bot/main.py`.

**Что прислать:**
- Точный **текст ошибки** на экране мини-аппа.
- С **какой** ссылки открывается (Vercel raw vs `?api=`).

**Этап:** воспроизвести цепочку Happ → возврат → сбросить кэш состояния / не терять `api` в URL / исправить редирект после добавления подписки.

## Приложение: типичные строки в `journalctl -u miniapp-api`

### Сканеры (не ошибка продукта)
Пачки запросов вида **`GET /.../.env`**, **`/.env.production`**, **`/wp-admin/setup-config.php`** с чужих IP — автоматический перебор. Ответ **404** нормален. При желании режьте **fail2ban** / **nginx limit_req** по таким путям.

### Happ `add-install` и 404 на `happ-proxy.com`
Эндпоинт **`/api/add-install`** должен вызываться с базы **`https://api.happ-proxy.com`**, не с корня **`https://happ-proxy.com`**. В коде добавлена **`resolve_happ_base_add_install()`** и подмена корня на **api.** (см. коммит после 2026-03-29).

### `local variable 'os' referenced before assignment` в `miniapp_me`
Исправлено: не делать **`import os`** внутри **`except`** в большой функции — иначе Python считает **`os`** локальной для всей функции.

### ЮKassa `Read timed out`
Увеличены таймауты HTTP к **api.yookassa.ru** до **30** с в **`bot/utils/payments.py`**.
