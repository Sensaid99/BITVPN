# Что деплоится и куда (чтобы не оставалась старая версия)

Кратко: какие файлы и куда попадают при деплое, и в каком порядке запускать скрипты.

---

## Что в репозитории (GitHub)

При **git add -A** и **git push** в репозиторий уходят **все** файлы проекта, кроме указанных в `.gitignore`:

- **Исключено:** `.env`, `.env.local`, `*.db`, `venv/`, логи, кэш Python.
- **Входит в репозиторий:** весь код бота и API, папки **deploy/** (скрипты nginx, apply-nginx-sub.sh, check-sub-url.sh, nginx-https-by-ip.conf, сервисы systemd), **docs/**, **webapp/**, **public/**, **api/**, конфиги, батники деплоя, **deploy_config.cmd** (без секретов).

Перед деплоем убедитесь, что все нужные правки сохранены и при необходимости сделаны **git add -A** и **git commit**, чтобы в пуш попала актуальная версия.

---

## Деплой на сервер (155.212.164.135)

**Скрипт:** `ДЕПЛОЙ_НА_СЕРВЕР.bat` (или шаг 1 в `ДЕПЛОЙ_ВСЁ.bat`).

Что происходит по шагам:

1. **Синхронизация:** копирование `webapp\index.html` → `public\index.html` → `index.html`, `api\root_index.html` (чтобы на сервер и в репозиторий шла одна версия мини-аппа).
2. **Git:** `git add -A`, `git status --short`, `git commit -m "Deploy"`, `git push origin main` (ветка из **deploy_config.cmd**, обычно `main`). В GitHub попадают все изменения, включая **deploy/** и **docs/**.
3. **Копирование .env:** если в **deploy_config.cmd** задано `COPY_ENV_TO_SERVER=1`, файл `.env` с вашего ПК копируется на сервер в `%BOT_PATH%/.env` (например `/opt/vpn-bot/.env`).
4. **На сервере по SSH:** переход в `BOT_PATH`, `git fetch origin`, `git checkout main`, `git reset --hard origin/main` (подтягивается актуальный код из GitHub), перезапуск **vpn-bot** и **miniapp-api**, запуск **deploy/apply-nginx-sub.sh** (добавление/проверка `location /sub/` в nginx).

Итог: на сервере — последний код из репозитория, актуальный `.env` (если копировали), nginx с поддержкой `/sub/`. Старая версия кода на сервере перезаписывается за счёт `git reset --hard origin/main`.

## Деплой мини-аппа на Vercel

**Скрипт:** `ДЕПЛОЙ_МИНИАПП_VERCEL.bat` (или шаг 2 в `ДЕПЛОЙ_ВСЁ.bat`).

Что происходит:

1. При необходимости синхронизация **public** → **webapp** (если правили public).
2. В папке **webapp** выполняется `npx vercel --prod`. На Vercel выкатывается содержимое папки **webapp** (то, что у вас локально в webapp).

Чтобы на Vercel не оставалась старая версия: перед запуском этого батника убедитесь, что в **webapp** лежит актуальная версия (при использовании **ДЕПЛОЙ_ВСЁ.bat** сначала выполняется деплой на сервер, где синхронизация webapp → public уже сделана; при отдельном запуске Vercel править нужно в **webapp** или сначала скопировать туда из public).

---

## Деплой «всё сразу»

**Скрипт:** `ДЕПЛОЙ_ВСЁ.bat`.

Порядок:

1. **Синхронизация:** webapp → public → index.html, api/root_index.html (одна версия мини-аппа для репозитория и сервера).
2. **Деплой на сервер:** пуш в GitHub, копирование .env, обновление кода на сервере (git pull + reset), перезапуск бота и API, apply-nginx-sub.sh.
3. **Деплой на Vercel:** `npx vercel --prod` из папки **webapp**.

Так везде оказывается актуальная версия: GitHub, сервер и Vercel.

---

## Файлы деплоя на сервере (папка deploy/)

После **git pull** на сервере в `/opt/vpn-bot/deploy/` лежат:

- **apply-nginx-sub.sh** — добавление `location /sub/` в nginx (вызывается из батника деплоя).
- **check-sub-url.sh** — диагностика, почему ссылка /sub/ не открывается (запуск вручную на сервере).
- **nginx-miniapp-api.conf** — образец блоков location для /sub/ и /api/.
- **nginx-https-by-ip.conf** — образец server-блока для HTTPS по IP (для ручной настройки по инструкции).
- **vpn-bot.service**, **miniapp-api.service** — юниты systemd (копируются вручную при первой настройке).

Все эти файлы должны быть в репозитории (не в .gitignore), тогда при деплое они подтягиваются на сервер вместе с кодом.

---

## Настройки (deploy_config.cmd)

- **SERVER_IP**, **BOT_PATH**, **GIT_BRANCH** — куда и откуда деплоить.
- **COPY_ENV_TO_SERVER=1** — при каждом деплое копировать локальный `.env` на сервер (чтобы не оставался старый .env).
- Локальный **.env** не попадает в Git (в .gitignore); на сервере после копирования лежит актуальный .env с вашего ПК.

---

## Краткий чеклист «нигде не деплоить старое»

1. Сохранить все изменения в проекте (файлы, deploy/, docs/).
2. Запустить **ДЕПЛОЙ_ВСЁ.bat** (или сначала **ДЕПЛОЙ_НА_СЕРВЕР.bat**, затем **ДЕПЛОЙ_МИНИАПП_VERCEL.bat**).
3. Убедиться, что пуш в GitHub прошёл (в окне батника нет ошибки push).
4. На сервере после деплоя код обновлён через **git reset --hard origin/main**; при **COPY_ENV_TO_SERVER=1** на сервере лежит свежий .env.
5. Vercel обновлён последним содержимым папки **webapp** после запуска ДЕПЛОЙ_МИНИАПП_VERCEL (или шага 2 в ДЕПЛОЙ_ВСЁ).

Если нужно, чтобы какой-то новый файл (например, в deploy/ или docs/) точно попал на сервер и в GitHub — добавьте его в проект и сделайте деплой через **ДЕПЛОЙ_НА_СЕРВЕР.bat** или **ДЕПЛОЙ_ВСЁ.bat**; **git add -A** подхватит все изменения.
