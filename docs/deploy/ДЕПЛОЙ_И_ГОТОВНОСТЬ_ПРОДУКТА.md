# Деплой и готовность продукта

Ниже: **как задеплоить** изменения и **чек-лист**, чтобы продукт был полностью готов. Проверки по коду выполнены.

---

## Где сейчас стоит мини-апп

- **Интерфейс (HTML/JS)** отдаётся с **Vercel**: у вас в `.env` указано `WEBAPP_URL=https://bitvpn.vercel.app?api=...` — пользователь открывает именно этот адрес. Запросы к API уходят на ваш сервер (`?api=https://nikolay.lisobyk.fvds.ru`).
- **Бот и API** крутятся на **вашем сервере** (николай.lisobyk.fvds.ru / IP из `deploy_config.cmd`).

**Нужно ли обновлять мини-апп на Vercel?**  
— **Нет**, если вы меняли только бота или API (логику на сервере). Достаточно деплоя на сервер и перезапуска API.  
— **Да**, если вы меняли интерфейс (файлы в `webapp\` или `public\index.html`) — тогда нужно задеплоить на Vercel (`ДЕПЛОЙ_МИНИАПП_VERCEL.bat` или push при автодеплое).

---

## Почему в мини-аппе ничего не меняется

**Мини-апп открывается с Vercel** (у вас в `.env`: `WEBAPP_URL=https://bitvpn.vercel.app?api=...`). То есть HTML отдаёт **не** ваш сервер, а проект на Vercel.

- **`ДЕПЛОЙ_НА_СЕРВЕР.bat`** обновляет только **сервер**: пушит в GitHub и по SSH подтягивает код на сервер, перезапускает бота. Сам по себе этот скрипт **не заливает** новый код на Vercel.
- Vercel обновляется только если проект на Vercel подключён к вашему GitHub, включён автодеплой при push и в настройках проекта указана **Root Directory: `webapp`**.

Чтобы изменения интерфейса появились на bitvpn.vercel.app, нужно **деплоить мини-апп на Vercel** (см. ниже).

---

## Как правильно деплоить после каждых изменений

**Источник мини-аппа в скриптах — папка `webapp`.** Скрипты копируют `webapp\index.html` → `public\index.html` → `index.html` и `api\root_index.html`.

### Если меняли только бота или API (мини-апп не трогали)

1. Запустите **`ДЕПЛОЙ_НА_СЕРВЕР.bat`** — обновится сервер и пуш в GitHub.
2. Если API на том же сервере — перезапустите: `ssh root@ВАШ_IP "sudo systemctl restart miniapp-api"`

### Если меняли мини-апп — чтобы изменения появились на bitvpn.vercel.app

**Вариант А — деплой мини-аппа на Vercel вручную (рекомендуется, сразу видно результат):**

1. Убедитесь, что правки есть в **`webapp\index.html`**. Если правили только `public\index.html`, скопируйте один раз:  
   `copy /Y public\index.html webapp\index.html`
2. Запустите **`ДЕПЛОЙ_МИНИАПП_VERCEL.bat`** (двойной клик).  
   Скрипт задеплоит папку `webapp` на Vercel (`npx vercel --prod`). В конце будет URL — откройте приложение из бота и проверьте.
3. При необходимости затем запустите **`ДЕПЛОЙ_НА_СЕРВЕР.bat`**, чтобы синхронизировать public/index/api и обновить сервер.

**Вариант Б — через GitHub (если Vercel подключён к репо и автодеплой включён):**

1. Редактируйте **только `webapp\index.html`** (или скопируйте public → webapp).
2. Запустите **`ДЕПЛОЙ_НА_СЕРВЕР.bat`** или **`ПУШ_НА_ГИТХАБ.bat`** — скрипт скопирует webapp → public → index, api и сделает push.
3. В [vercel.com](https://vercel.com) → ваш проект → Deployments: должен появиться новый деплой. Если нет — используйте **Вариант А**.

**Кратко:** чтобы в мини-аппе что-то менялось, нужно деплоить **мини-апп на Vercel** — вручную **`ДЕПЛОЙ_МИНИАПП_VERCEL.bat`** или через push при настроенном автодеплое. Редактировать лучше **`webapp\index.html`**.

## Как задеплоить (подробно)

### Вариант 1: С вашего ПК (рекомендуется)

1. **Проверьте `deploy_config.cmd`** в корне проекта:
   - `SERVER_USER` — пользователь SSH (например `root`);
   - `SERVER_IP` — **IP** сервера, куда идёт SSH (например `155.212.164.135`). Должен быть тот же сервер, на котором крутится API по домену из `WEBAPP_URL` (у вас `?api=https://nikolay.lisobyk.fvds.ru` — домен должен указывать на этот IP);
   - `BOT_PATH` — каталог с ботом на сервере (например `/opt/vpn-bot`);
   - `RESTART_CMD` — команда перезапуска бота (например `sudo systemctl restart vpn-bot`).

2. **Запустите деплой:**
   - Дважды щёлкните **`ДЕПЛОЙ_НА_СЕРВЕР.bat`**  
   или в командной строке из папки проекта: `ДЕПЛОЙ_НА_СЕРВЕР.bat`

3. Скрипт сделает:
   - синхронизацию `webapp` → `public` → `index.html`, `api/root_index.html`;
   - `git add` + `commit` + `push` в текущую ветку (часто `main`);
   - по SSH на сервер: `cd BOT_PATH && git fetch && git reset --hard origin/ветка && RESTART_CMD` и **перезапуск miniapp-api** (чтобы API подхватил новый код).

4. **Если на том же сервере крутится API** (например `api_miniapp.py` через uvicorn или systemd), после деплоя перезапустите и его:
   ```bash
   ssh root@ВАШ_IP "sudo systemctl restart miniapp-api"
   ```
   (имя сервиса может быть другим — как вы его назвали при настройке.)

5. **`.env` на сервер скрипт не копирует** (чтобы не затереть настройки). Если меняли переменные в локальном `.env` и хотите их на сервере — скопируйте вручную или добавьте в скрипт строки для `scp .env`.

---

### Вариант 2: Вручную на сервере

Если деплой с ПК не используете:

```bash
ssh root@ВАШ_IP
cd /opt/vpn-bot
git fetch origin
git reset --hard origin/main
sudo systemctl restart vpn-bot
# Если API на этом же сервере:
sudo systemctl restart miniapp-api
```

После этого на сервере будет актуальный код (включая вебхук ЮKassa и «Подключено X/Y» в профиле).

## Что проверено в коде

- **Оплата и лимит устройств:** при создании ссылки Happ в боте и в вебхуке передаётся `install_limit` из тарифа (`1_month_1` → 1, `6_months_3` → 3 и т.д.). Ограничение по числу устройств обеспечивает Happ.
- **Вебхук ЮKassa:** `POST /api/webhook/yookassa` в `api_miniapp.py` обрабатывает `payment.succeeded`, находит платёж по `object.id`, вызывает `_complete_payment_and_send_link`, создаёт подписку, получает ссылку Happ (или конфиг), отправляет её в Telegram, возвращает 200.
- **Профиль «Подключено X/Y»:** в `show_profile` при активной подписке и наличии `installid` в `vpn_config` вызываются `parse_install_code_from_happ_link` и `get_install_stats` (Happ API `list-install`), в текст профиля добавляется строка «Подключено устройств: X/Y».
- **Защита от двойной выдачи:** при нажатии «Проверить платеж» для уже выполненного платежа (`status == 'completed'`) подписка не создаётся повторно, показывается сообщение «Оплата уже проведена…».

## Если что-то не работает

- **Ссылка после оплаты не приходит:** проверить логи API на сервере (запросы на `/api/webhook/yookassa`, ошибки при отправке в Telegram). Убедиться, что в ЮKassa указан правильный URL и включён `payment.succeeded`.
- **В профиле нет «Подключено X/Y»:** проверить, что в .env бота заданы `HAPP_PROVIDER_CODE` и `HAPP_AUTH_KEY`; что у подписки в БД в `vpn_config` хранится ссылка с `installid=...`.
- **Мини-апп не открывается или не создаёт платёж:** проверить `WEBAPP_URL` и то, что мини-апп обращается к вашему API (переменная `api` / MINIAPP_API_URL), и что API доступен по HTTPS.

После выполнения чек-листа и успешного деплоя продукт готов к использованию.
