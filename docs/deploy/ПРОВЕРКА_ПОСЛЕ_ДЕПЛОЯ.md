# Проверка после деплоя

## Если 404 в Telegram при нажатии «Открыть VPN»

- **BotFather:** @BotFather → `/setmenubutton` → выберите бота → Web App → URL: `https://bitvpn.vercel.app?api=https://bitecosystem.ru` (как в `WEBAPP_URL` на сервере), текст: «Открыть VPN».
- **.env на сервере:** `WEBAPP_URL=https://bitvpn.vercel.app?api=https://bitecosystem.ru`, `MINIAPP_API_URL=https://bitecosystem.ru`.
- **Перезапуск:** после смены .env на сервере: `sudo systemctl restart vpn-bot`. Полностью закройте Telegram и откройте снова или удалите чат с ботом и начните заново.

---

## Если 404 у API (check-happ-env и т.п.)

Если после деплоя `https://bitecosystem.ru/api/miniapp/check-happ-env` по-прежнему возвращает `{"detail":"Not Found"}`, выполните проверки **на сервере** по шагам.

---

## Шаг 1. Подключитесь к тому же серверу, куда идёт деплой

Деплой из `deploy_config.cmd` идёт на **SERVER_IP** (у вас `213.165.38.222`). Подключитесь именно к нему:

```bash
ssh root@213.165.38.222
```

(Пароль тот же, что вы вводите при запуске ДЕПЛОЙ_НА_СЕРВЕР.bat.)

---

## Шаг 2. Есть ли маршрут в файле на сервере

Выполните на сервере:

```bash
grep -n "check-happ-env" /opt/vpn-bot/api_miniapp.py
```

- **Если выводится строка** (например `286:@app.get("/api/miniapp/check-happ-env")`) — код с маршрутом на сервере есть. Переходите к шагу 3.
- **Если пусто** — на сервере старая версия кода. Убедитесь, что вы пушите в GitHub с ПК (в папке проекта выполните `git status`, `git log -1 --oneline` и при необходимости `git push origin main`), затем снова запустите ДЕПЛОЙ_НА_СЕРВЕР.bat.

---

## Шаг 3. Отвечает ли сам API (uvicorn) без nginx

На сервере выполните:

```bash
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8765/api/miniapp/check-happ-env
```

- **Если вывод `200`** — приложение на сервере отдаёт маршрут. Тогда 404 снаружи чаще из‑за nginx или DNS: **bitecosystem.ru** должен указывать на **213.165.38.222**. Проверьте: `https://bitecosystem.ru/api/miniapp/check-happ-env` или по IP с `-k`: `curl -k https://213.165.38.222/api/miniapp/check-happ-env`.
- **Если вывод `404`** — процесс API запущен со старым кодом. Перезапустите API:
  ```bash
  sudo systemctl restart miniapp-api
  ```
  Подождите 2–3 секунды и снова выполните команду `curl` из шага 3. Должно стать 200.

---

## Шаг 4. Один ли сервер у домена и у деплоя

На своём ПК (PowerShell или cmd) выполните:

```bash
ping -n 1 bitecosystem.ru
```

Посмотрите IP в ответе. Если там **не** `213.165.38.222`, то при открытии в браузере bitecosystem.ru запросы идут на другой хост, куда деплой не делается. Нужно либо деплоить и на тот сервер, либо настроить домен на `213.165.38.222`.

---

## Кратко

| Проверка | Команда / действие |
|----------|---------------------|
| Код на сервере | `grep -n "check-happ-env" /opt/vpn-bot/api_miniapp.py` |
| API отвечает локально | `curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:8765/api/miniapp/check-happ-env` |
| Перезапуск API | `sudo systemctl restart miniapp-api` |
| Куда смотрит домен | `ping bitecosystem.ru` — должен быть 213.165.38.222 |

После выполнения пришлите вывод этих команд — по ним можно точно сказать, на каком шаге проблема.
