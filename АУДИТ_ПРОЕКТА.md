# Аудит проекта VPN BOT (Bit VPN)

Дата: март 2025.

---

## 1. Что уже сделано

### Бот и запуск
- Точки входа: `run.py`, `start_bot.py` → `bot.main.main()`.
- Конфиг: `bot/config/settings.py`, переменные из `.env`.
- Обработчики: старт, профиль, конфиг, рефералы, помощь, поддержка, вывод средств (conversation), админка, рассылка.
- Кнопка Mini App: `MenuButtonWebApp` при наличии `WEBAPP_URL`.
- Глобальный обработчик ошибок в боте.

### Mini App API (`api_miniapp.py`)
- FastAPI, CORS, валидация `initData` (HMAC с BOT_TOKEN).
- Эндпоинты: `GET /`, `GET /health`, `POST /api/miniapp/me`, `GET /api/miniapp/plans`, `POST /api/miniapp/create-payment`.
- Интеграция с БД и платёжным менеджером (ЮKassa и др.), создание платежа и возврат `payment_url`.

### Фронт (Mini App)
- Один HTML/JS файл (`webapp/index.html`, копии в `public/`, `api/`).
- Работа с API по параметру `?api=...`, вызовы `/me`, `/plans`, `/create-payment`, открытие ссылки оплаты.

### Деплой
- systemd: `vpn-bot.service`, `miniapp-api.service`, `support-bot.service`.
- Nginx: конфиг с HTTP/HTTPS и `location /api/`, certbot, симлинк `sites-enabled/default`.
- Инструкция: `ЛОГИ_API_НА_СЕРВЕРЕ.txt`, автообновление сертификата (certbot.timer).
- Vercel: `vercel.json`, `api/` для серверного API (опционально).

### Документация
- README, ЗАПУСК, СИНХРОНИЗАЦИЯ_БОТ_МИНИАПП, HAPP_*, ЮKASSA_ИНТЕГРАЦИЯ и др.

### Проверки
- `test_setup.py`, `test_bot.py`, `validate_bot.py` — ручные/проверочные скрипты.

## 3. Что стоит добавить (рекомендации)

### Безопасность
- **Ограничить CORS в API:** вместо `allow_origins=["*"]` указать конкретные источники (например, `https://bitvpn.vercel.app`, `https://nikolay.lisobyk.fvds.ru`). Так мини-апп по-прежнему будет работать, а запросы с левых сайтов — нет.
- **Rate limit на создание платежа:** ограничить число запросов `POST /api/miniapp/create-payment` с одного IP/пользователя (например, 5–10 в минуту), чтобы снизить риск злоупотреблений и брутфорса.

### Надёжность и отладка
- **Request ID в API:** добавлять к каждому запросу уникальный `X-Request-ID` и прокидывать его в логи. Упростит поиск по логам при ошибках оплаты.
- **Структурированное логирование:** в ключевых местах (create-payment, ошибки БД/ЮKassa) логировать JSON с полями `request_id`, `telegram_id`, `error`, чтобы удобнее парсить логи (например, через journalctl или сборщики логов).

### Документация и разработка
- **OpenAPI/Swagger для API:** FastAPI умеет отдавать `/docs` и `/openapi.json`. Сейчас не отключено, но и не упомянуто в инструкциях — можно явно описать в README или в `ЛОГИ_API_НА_СЕРВЕРЕ.txt`, что после деплоя API доступна документация по `https://ваш-домен/docs`.
- **Краткий «Переменные окружения»:** в README или отдельном файле перечислить все переменные из `bot/config/settings.py` и `api_miniapp` (BOT_TOKEN, DATABASE_URL, YOOKASSA_*, WEBAPP_URL, MINIAPP_API_URL и т.д.) с кратким описанием — проще настраивать новый сервер.

### Тесты
- **Автотесты API:** минимальный набор (pytest + httpx): проверка `GET /api/miniapp/plans` без авторизации, проверка `POST /api/miniapp/me` с невалидным/валидным `initData`, при желании — мок create-payment. Это защитит от поломок при изменениях в `api_miniapp.py`.

### Фронт (Mini App)
- **Обработка сетевых ошибок:** при недоступности API показывать пользователю понятное сообщение («Проверьте интернет» / «Сервер временно недоступен») вместо пустого экрана или технической ошибки.
- При желании — вынести список эндпоинтов и версию API в константы в одном месте, чтобы при смене домена/версии не искать по всему коду.

## 4. Советы по приоритетам

1. **Сделать в первую очередь:** корневой `.env.example` и пункт про `sites-enabled` в инструкции — быстро и снижает ошибки при развёртывании.
2. **Далее:** сузить CORS и добавить rate limit на create-payment — мало кода, заметный плюс по безопасности.
3. **По мере развития:** request_id + структурированные логи, затем автотесты API и явное описание переменных окружения.
